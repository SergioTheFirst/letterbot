# MailBot Premium — Strategy (v1 → v1.5)

## 0. Контекст и цель
MailBot достиг состояния “инженерная система, а не скрипт”:
- детерминированный пайплайн;
- наблюдаемость и метрики;
- штатные деградации;
- events как память;
- стабильный UX в Telegram;
- RU-first;
- single-flight LLM;
- зелёные тесты как инвариант.

Цель стратегии — не “добавлять фичи”, а зафиксировать MailBot как локальный когнитивный слой над почтой:
- быстро информирует;
- накапливает ценность (контекст/паттерны/рекомендации);
- не ломается при деградациях;
- остаётся объяснимым.

## 1. Ключевой стратегический выбор
### Fast First, Deep Later, Deterministic Core
**Правило:** любое письмо должно быть корректно и безопасно доставлено пользователю как можно раньше.  
Обогащение контекстом, паттернами и рекомендациями выполняется асинхронно и условно.

**Почему не “LLM first”:**
- увеличивает латентность;
- ломается при деградации LLM;
- ухудшает UX (пользователь ждёт, не понимая, что пришло);
- снижает объяснимость.

**Почему не “только правила”:**
- не накапливает ценность;
- не даёт wow-эффекта;
- не масштабируется по интеллекту.

**Выбранный путь:**
- rules-first → мгновенный объяснимый сигнал;
- events-first → память и аналитика;
- LLM-later → усилитель, а не SPOF.

## 2. Архитектурные основы (фундамент)
### 2.1 Events as Source of Truth
**Правило:** аналитика, паттерны, доверие, качество и SLA читаются **только** из `events_v1`.

**Зачем:**
- исключает “double truth”;
- делает поведение воспроизводимым;
- позволяет пересчёт/миграции;
- поддерживает деградации (events пишутся даже без LLM/TG).

**Почему не читать напрямую из emails:**
- emails — сырьё;
- events — интерпретация;
- аналитика зависит от интерпретации, а не от сырья.

### 2.2 Deterministic Core, LLM as Enhancement
**Правило:** классификация, приоритет, нарратив, SLA и качество — детерминированы.  
LLM:
- не меняет порядок пайплайна;
- не меняет Telegram schema;
- не используется, если rules дают достаточный сигнал.

**Причины:** тестируемость, объяснимость, воспроизводимость, эксплуатационная и юридическая безопасность.

### 2.3 Single-flight LLM (жёсткий инвариант)
**Решение:** единый ingress и глобальный lock для LLM (single-flight).  
Любая попытка параллелизации/обхода запрещается тестами.

**Причины:** требования провайдера, предсказуемая латентность, отсутствие “шипов” нагрузки, простая деградация вместо гонок.

## 3. UX-стратегия (Telegram-first)
### 3.1 Telegram — продукт, не лог
Telegram-сообщение — готовая когнитивная единица:
- Premium Formatter;
- строгая HTML-safety;
- salvage delivery;
- отсутствие бинарного мусора;
- RU-first тексты;
- explainability (“ПОЧЕМУ ТАК”).

**Причина:** пользователь читает продукт, а не дебажит.

### 3.2 Fast First ≠ “всегда 5 секунд”
Скорость не достигается ценой качества:
- не задерживаем, если можем не задерживать;
- простые письма: 0–5 сек;
- wow-результаты: 30–180 сек;
- при деградации: всегда корректно.

## 4. Приоритезация и доверие
### 4.1 Priority Engine v2 — shadow-first
**Правило:** сначала shadow-режим; auto — только после прохождения gates PASS.  
Все коррекции → в events → метрики качества.

**Причины:** защита от регрессий, измеряемый прогресс, user-in-the-loop без риска.

### 4.2 Trust Score v2 — temporal, forgiving (rule-based)
Свойства:
- экспоненциальный decay;
- “redemption arc” встроен;
- нет вечных наказаний;
- полностью rule-based.

**Причины:** соответствует реальности отношений, устойчив к шуму, работает без LLM.

## 5. Надёжность и эксплуатация
### 5.1 Graceful Degradation — норма
Любой компонент может быть недоступен: IMAP / Telegram / LLM / SQLite (busy).  
Ожидаемое поведение:
- система продолжает работать;
- пользователь информирован;
- события и логи фиксируются;
- recovery предусмотрен.

### 5.2 Doctor / SLA / Quality — обязательные контуры
- **Doctor:** read-only диагностика, гейтит запуск.
- **SLA:** latency, salvage rate, error breakdown.
- **Quality:** измеряем, а не “верим”.

## 6. Интернационализация
- RU-first UI;
- внутренние коды — английские;
- humanizers на границе UX.

## 7. Что сознательно НЕ делаем сейчас
- **Web UI:** нет bottleneck; Telegram покрывает 95% сценариев; высокий риск распыления.
- **Embeddings / ML:** нарушает детерминизм; усложняет объяснимость; не окупается на этапе.
- **“умнее любой ценой”:** интеллект без надёжности = шум; wow без доверия = разочарование.

## 8. Вердикт
Это стратегия:
- инженерная, а не модная;
- воспроизводимая, а не “магическая”;
- ориентированная на пользователя, а не на демо.

MailBot — не LLM-продукт.  
MailBot — когнитивная система, где LLM — инструмент, а не мозг.
