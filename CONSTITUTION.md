Status: Active, binding
Scope: Product UX (Telegram), architecture, data, AI usage, reliability, privacy, development rules

I. Миссия проекта

MailBot Premium — это не почтовый клиент и не автоответчик.

MailBot — это локальный когнитивный слой поверх почты, который превращает письма в историю отношений, обязательств и доверия.

Система обязана:

- снижать когнитивную нагрузку пользователя;
- выявлять скрытые риски и паттерны;
- накапливать ценность со временем (memory compounds);
- работать надёжно даже при деградации внешних сервисов;
- быть честной: не придумывать, а показывать проверяемые факты и аккуратные выводы.

II. Базовые ограничения (НЕ обсуждаются)
1) Почти бесплатно

Разрешены только:

- Cloudflare Workers AI
- GigaChat (Sber)

Запрещено без отдельного решения:

- любые платные SaaS/подписки/внешняя GPU-инфраструктура.

2) Windows-first

Система обязана работать на обычном Windows-ПК.
Запрещено: Docker-only, Linux-only, Kubernetes-зависимости как обязательное условие.

3) Offline-first мышление

Внешние сервисы — усиление, не опора.
При недоступности LLM система продолжает работать в ограниченном режиме (без «падения целиком»).

4) Данные принадлежат пользователю

- локальная БД: SQLite;
- прозрачный экспорт;
- никакого vendor lock-in.

III. UX Конституция: Premium-опыт в Telegram
1) Premium = мгновенная ясность

Пользователь должен за 2–3 секунды понять:

- кто пишет,
- что хотят,
- когда важно,
- что делать дальше.

2) One-Message Rule (строгая)

Одно письмо → одно пользовательское уведомление в Telegram. Никогда два.

Допускаются только исключения:

- Операционные алерты о деградации (IMAP/DB/Telegram/LLM): агрегировано, без спама, без дублей.
- Ответ на действие пользователя (on-demand), включая команду /details… и ровно одну кнопку «▶ Подробнее».

Правило «Подробнее»

- Tier-2 показывается без второго push: через редактирование того же сообщения (edit-in-place) и/или через ответ на команду пользователя.
- Допускается UI-вариант: одна кнопка «▶ Подробнее» внизу уведомления.
- Кнопка «▶ Подробнее» не создаёт новое уведомление сама по себе; она либо редактирует исходное сообщение, либо отвечает на команду пользователя.
- Спойлер без взаимодействия невозможен для длинных блоков: длинные детали показываются только после действия пользователя.

3) Trust but verify: факты обязаны быть видимыми

Вывод без фактов разрушает доверие и опасен (в т.ч. security/anti-phishing).

Tier-1 всегда включает проверяемые факты (если доступны):

- От: (имя + email/домен),
- Тема: subject,
- Вложения: имена файлов (до лимита) + короткий статус извлечения/содержания.

Правило краткости: если информации нет — не пишем «нет данных», блок просто отсутствует.

4) Progressive disclosure: Tier-1 / Tier-2 / Tier-3

- Tier-1 (real-time, всегда): «Факты → Вывод → Действие». Коротко, RU-first, без внутренних enum/кодовых причин в тексте.
- Tier-2 (on-demand): подробности по конкретному письму/треду: контекст, история, «почему», риски, рекомендации, опциональные шаблоны. В Telegram — только через кнопку «▶ Подробнее» (edit-in-place) и/или /details. Спойлер допустим только для коротких инлайн-фрагментов; длинные детали требуют явного действия.
- Tier-3 (digest/аналитика): агрегированные инсайты и аномалии только если есть смысл (no empty digests).

5) Trust dots (индикатор уверенности)

- Используется точечный индикатор из точек/кружков без текста причин; всего 5 уровней.
- Показывается только когда есть вычисленная уверенность; по умолчанию отображается при confidence ≥ 0.75 (75%).
- В Tier-1 не показывать внутренние enum/причины; только сами точки.
- RU-first подписи; никакой смешанной локализации.

6) Формат Tier-1 (жёсткий стандарт)

- Лимит строк: целевой 8–12, максимум 15.
- Вложения: показывать до 2 штук полностью; если >3, показываем первые 2 и строку «… и ещё N».
- Структура: От → Тема → Вывод/риск → Действие → (опционально) доверительные точки/уверенность → файлы/статусы.
- Никаких избыточных пояснений и внутреннего шума.

7) Emoji-словарь (закрытый)

См. Приложение A. Любые эмодзи вне списка запрещены и должны быть удалены из Telegram-вывода.

IV. Архитектура и надёжность

- Fast-first: уведомление Tier-1 не ждёт долгих извлечений/LLM; обновления допускаются только как edit-in-place.
- Детерминированный SLA доставки: ждём до MAX_TELEGRAM_WAIT_SECONDS для полного Tier-1; при превышении — отправляем минимальный Tier-1 один раз и обогащаем только через edit-in-place.
- Нет тихих часов/выходных/таймзонных задержек для уведомлений: доставка всегда немедленная по фактам.
- Режим доставки ограничен IMMEDIATE; shadow-аналитика не влияет на факт отправки; batching/deferral запрещены, любое обогащение только через edit-in-place в рамках SLA.
- Разделение worker/web: обработка писем и Telegram-поток не должны блокировать друг друга.
- Телеметрия и логи — без утечек PII за пределы контролируемых хранилищ.
- Веб-консоль наблюдаемости только для чтения; отображает обезличенные снапшоты состояния (gates/metrics/system_mode) без payload_json.
- Web = read-only Intelligence Hub; events timeline renders sanitized summaries only; never show raw email text/subjects/senders in web.
- Relationship graph и прочие веб-аналитики всегда только чтение, поверх обезличенных снапшотов и агрегатов; никаких сырых тел/HTML/вложений.
- Интеллект-страницы (health/events/relationships/attention) отображают только агрегированные метрики без PII и не модифицируют БД.
- Learning Observatory (web + API) — только чтение, детерминировано, строится из events_v1, с PII-санитайзером; shadow-классификация не меняет delivery defaults.

V. Privacy и данные

- Личные данные пользователя не покидают локальный узел без явного запроса.
- Сырые письма и вложения не уходят в облако; для LLM допускается только минимально необходимый контекст.
- Экспорт всегда возможен.

VI. AI (LLM) правила

- Cloudflare / Sber GigaChat — единственные разрешённые LLM.
- Sber GigaChat: строго один поток. Глобальный lock обязателен и не может быть отключён. При занятости/ошибке — фолбэк в Cloudflare или деградированный режим без блокировки пайплайна.
- Никаких «галлюцинаций ради полноты»: LLM-вывод должен ссылаться на факты.
- Любое обучение/адаптация — только через события и метрики с возможностью отката.

VII. Events и память

- Events — вечная память решений. Сырые email body/subject/telegram_text/rendered_message/digest_text в событиях НЕ хранятся.
- Хранить выводы + опорные факты + уверенность обязательно.
- Изменения эвристик/порогов — только через shadow-валидацию, метрики и откат.

VIII. Runtime-инварианты: надёжность, минимальные зависимости

- Без тяжёлых зависимостей, никакого LibreOffice.
- Лёгкие извлекатели: pdftotext, catdoc, xlhtml, xlsx2csv и т.п.
- Строгая идемпотентность и IMAP UID дисциплина: первичный SEARCH UID 1:*, max_uid как last_uid, предотвращение дублей, atomic state writes, UID-лог.
- Ingest по умолчанию — только post-start (письма, поступившие после старта процесса).
- Backfill допускается только при явном maintenance-конфиге.
- Разделение worker и web/UX процессов.

IX. Development Rules

- Если фича не укладывается в архитектуру — она не делается.
- Side-effects изолированы: Telegram/логи/БД не должны ломать pipeline.
- Каждое решение объяснимо из кода и логов.
- Не внедрять улучшения в мёртвые пути: изменения либо на runtime-пути, либо под флагом, который реально маршрутизирует runtime.
- Тесты обязательны (инкрементально).
- CONTINUITY.md обновлять одной короткой строкой в конце серии изменений, без лишнего текста.

X. Roadmap Alignment

- v26–v27: фундамент (observability, events, trust/health, graceful degradation)
- v28: поведенческие паттерны
- v29: attention economics
- v30: predictive intelligence
- Никакой «магии» без данных, метрик и гейтов.

XI. Финальный принцип

MailBot — это система, которая со временем начинает понимать не письма, а людей и их надёжность.

Если новая фича:

- не усиливает контекст,
- не накапливает память,
- не повышает доверие,

→ она противоречит Конституции и не принимается.
